- **Task 1.1**: Scaffolded with `create-next-app` (Next 16, React 19, Tailwind v4, Turbopack, App Router, no `src/` dir, `@/*` import alias). shadcn/ui initialised with the `neutral` base via `--defaults`. Home page lives at `app/(guest)/page.tsx`; root layout in `app/layout.tsx` wraps all routes with `Header`/`Footer` from `components/layout/`. Prettier configured with `prettier-plugin-tailwindcss`. `.env.example` committed; `.env.local` gitignored.
- **Task 2.7**: Webhook handler at [app/api/webhooks/stripe/route.ts](app/api/webhooks/stripe/route.ts). Verifies the signature against `STRIPE_WEBHOOK_SECRET` using the **raw request body** (`request.text()` — Next.js JSON parsing would mutate it). Idempotency via a new [`WebhookEvent`](prisma/schema.prisma) table keyed on the Stripe event id: insert-or-fail-200 means duplicate deliveries are no-ops. On `checkout.session.completed` the handler calls [`confirmBooking`](lib/booking/confirm.ts) which atomically marks the Booking `confirmed` + writes a `Payment` row with the charge details (card brand/last4 are pulled by expanding `latest_charge.payment_method_details.card` on the PaymentIntent). On `checkout.session.expired` it calls `cancelPendingBooking` (also reachable from the 2.10 cron sweep). Booking confirmation email is currently a stub at [lib/email/booking.ts](lib/email/booking.ts) that just logs — Task 2.8 swaps in Resend + React Email. **If the handler throws after inserting the WebhookEvent row, the row is rolled back** so Stripe's retry will re-process the event rather than skip it. **Local testing requires** `stripe listen --forward-to http://localhost:3000/api/webhooks/stripe` running alongside `npm run dev`, with the printed `whsec_...` value pasted into `.env.local` as `STRIPE_WEBHOOK_SECRET`. Verified end-to-end: pay with test card → webhook fires → booking moves pending→confirmed → payment row written → email stub logged.
- **Task 2.6**: Stripe Checkout integration via `stripe` 22.0.1. [lib/stripe/client.ts](lib/stripe/client.ts) — lazy server-side singleton. [lib/booking/create.ts](lib/booking/create.ts) — `createBooking` orchestrates the full happy path: inside `withPropertyLock` it re-validates availability, calculates the price, and atomically inserts a `Booking` (status `pending`) plus its `BookingPriceSnapshot`; AFTER releasing the lock it creates a Stripe Checkout Session with multi-line items (accommodation, cleaning, extra guests) and embeds `booking_id` in metadata so the webhook in 2.7 can find it. Stripe rejection rolls back the pending booking immediately rather than waiting for the cron sweep. **Pending bookings hold dates for 30 minutes** (`PENDING_TTL_MINUTES`) — same as the Stripe session `expires_at` and the cron sweep cadence in 2.10. `POST /api/bookings` — auth-gated, Zod-validated, returns `{ bookingId, checkoutUrl }`. `/booking/success` and `/booking/cancel` are simple landing pages; the success page deliberately does NOT mark the booking confirmed (the webhook does that, otherwise a malicious user could fake a confirmation by hand-crafting the URL). Calendar wired up: `isSignedIn` prop drives "Reserve" vs "Sign in to book". On Reserve, the client POSTs to the endpoint and `window.location.href = url` to Stripe. **Two Prisma 7 Json gotchas**: (a) `nightlyRates: NightlyRate[]` and (b) the cancellation policy snapshot literal both fail the `InputJsonValue` index-signature check — cast via `as Prisma.InputJsonValue` (or `as unknown as` for the array case). **Calendar UX iteration**: future taken dates now visually distinct from past dates — strikethrough grey for "blocked or someone else's booking", emerald-100 highlight for "your own pending/confirmed booking" (the API returns `{ blocked, mine }` separately when the user is signed in). The user's own booked dates are still disabled — no double-booking yourself.
- **Task 2.5**: Supabase Auth via `@supabase/ssr` 0.10.2. Helpers in [lib/auth/](lib/auth/): `server.ts` (server component / route handler client + `getCurrentUser()`), `browser.ts` (singleton browser client), `middleware.ts` (`updateSession` runs `auth.getUser()` to refresh tokens, then gates `/dashboard|/messages|/admin` for unauthed users and bounces signed-in users away from `/login|/signup`). **Next 16 rename**: the file convention is `proxy.ts` not `middleware.ts` — Next deprecated the name to disambiguate from Express middleware. Function export is `proxy`, not `middleware`. Server actions in [lib/auth/actions.ts](lib/auth/actions.ts): `signUpAction`, `signInAction`, `magicLinkAction`, `signOutAction`. **Signup creates a `Guest` row in the same transaction as the auth user**, using the auth user's UUID as the Guest id (the schema's intended pattern from 2.1). Email-confirmation behaviour is handled by branching on `data.session` — if confirmation is on, signUp returns no session and the form shows a "check your email" message; if off, it redirects straight to `/dashboard`. Sign-in form has both password and magic-link options. `/auth/callback` route exchanges the magic-link / confirmation code for a session. Header is now an async server component that shows "Hi, {firstName}" and a sign-out form when authed. **Side effect of the proxy**: every page is now `ƒ` dynamic (was `○` static) because `auth.getUser()` runs per-request — fine for now, the home page renders fast. **Local Supabase has email confirmation off** for dev convenience (per [supabase/config.toml](supabase/config.toml#L208)); the **hosted** project is currently in the same state for testing but the user has chosen to flip it back to required-confirmation before launch (set `Authentication → Email → Confirm email` in the Supabase dashboard). **Open follow-up**: the dashboard page itself (Phase 3) doesn't exist yet, so signing up redirects to a 404. The auth flow still works correctly — proxy authorisation, session refresh, sign-in/out are all wired.
- **Task 2.4**: [AvailabilityCalendar](components/property/availability-calendar.tsx) — client component built on shadcn's `Calendar` (react-day-picker v9 in `mode="range"`). Fetches `/api/availability/blocked` once on mount and turns the day list into a `Set` for O(1) disabled-day matcher. On range change, calls `/api/pricing` for a live breakdown — no debounce yet because the calculator is fast. Adults/Children dropdowns with totals capped at `maxGuests`; min-stay/max-stay enforced client-side too so the user gets immediate feedback before they click Reserve. **Two shadcn-default visual fixes worth remembering**: (a) the default `today` style uses `bg-muted` which is the same fill as `range_middle`/`range_end` — overrode to a subtle ring so today doesn't visually merge with selected ranges; (b) Tailwind v4 dropped the legacy `cursor: pointer` button default — restored on enabled day cells via `[&>button]:cursor-pointer`. Subtle gotcha: putting the cursor on the cell rather than the inner button gives disabled cells the pointer too, because shadcn `Button` has `disabled:pointer-events-none` and the cursor falls through to the parent. Reserve button is wired but non-functional — the booking flow lands in 2.6/2.9.
- **Task 2.3**: [`calculatePrice`](lib/pricing/calculate.ts) returns a `PriceBreakdown` shaped to match `BookingPriceSnapshot` 1:1, so the same object can be passed straight to the booking-creation step in 2.6/2.9 with no remapping. **Phase 2 is intentionally simple**: flat `baseNightlyRate` per night, cleaning fee, extra-guest fee for guests above `baseGuestCount`, no discounts/service fee/tax. The full rules engine (seasonal/day-of-week/last-minute/early-bird/length discounts) lands in Phase 7 — when it does, only the per-night rate-resolution loop changes; the breakdown shape stays the same. **Money is `Prisma.Decimal` end-to-end**: serialised to fixed-2 decimal strings at the JSON boundary to avoid float drift. Validation split: pricing rejects only `invalid_range`/`property_not_found`/`too_many_guests` — `min_stay`/`max_stay` are availability's job, so the calculator can be called even for short preview windows. `GET /api/pricing?checkIn&checkOut&adults&children` returns the breakdown. 16/16 smoke tests in [scripts/test-pricing.ts](scripts/test-pricing.ts).
- **Task 2.2**: Availability service in [lib/availability/](lib/availability/) — `checkAvailability` returns a discriminated union of `{available, numNights}` or `{available:false, reason, message}` with reasons `in_past`, `invalid_range`, `min_stay`, `max_stay`, `blocked`, `booked`. **Date semantics nailed down**: bookings use half-open `[checkIn, checkOut)` (back-to-back stays allowed); blocked dates use closed `[dateStart, dateEnd]`. Date helpers in [dates.ts](lib/availability/dates.ts) keep everything in UTC. [`withPropertyLock`](lib/availability/lock.ts) wraps a callback in a Postgres `pg_advisory_xact_lock(hashtextextended('property:<id>', 0))` so concurrent booking attempts on the same property serialise — used by booking creation, not by read-only checks. Two new endpoints: `GET /api/availability?checkIn&checkOut` and `GET /api/availability/blocked?from&to` (defaults to today + 365 days, returns flat day-string array combining blocked + active bookings). Zod-validated query params. **Lazy Prisma singleton**: refactored [lib/db/prisma.ts](lib/db/prisma.ts) to use a Proxy so the underlying client is constructed on first access, not at module load — fixes CLI scripts whose `import` statements get hoisted above their `dotenv` calls (was crashing `scripts/test-availability.ts` with "DATABASE_URL is not set"). Hand-rolled smoke test in [scripts/test-availability.ts](scripts/test-availability.ts) — 13 assertions covering happy path, range validation, min/max stay, blocked-date overlap with half-open semantics, booking overlap, back-to-back booking allowed, and lock helper execution. Re-runnable, self-cleaning. No test framework yet — revisit when assertion count grows.
- **Task 2.1**: Six new entities in [prisma/schema.prisma](prisma/schema.prisma) — `Guest`, `Booking`, `BookingPriceSnapshot`, `Payment`, `BlockedDate`, `PricingRule` — plus enums (`BookingStatus`, `BookingType`, `PaymentType`, `PaymentStatus`, `PricingRuleType`). Field definitions follow the data-model doc. Snapshot principle is enforced by storing `cancellation_policy_snapshot` (Json) on `Booking` and the full itemised price as a one-to-one `BookingPriceSnapshot`. **`Guest.id` has no `@default(uuid())`** — it's intended to mirror `auth.users.id` from Supabase Auth so a single UUID identifies a user across both schemas; the row is created at signup time using the auth user id. No `password_hash` field — credentials live entirely in Supabase Auth. Indexes added: `(propertyId, checkIn, checkOut, status)` on bookings (the hot availability query), `(propertyId, dateStart, dateEnd)` on blocked dates, `(propertyId, isActive, priority)` on pricing rules. Migration `20260412171311_add_booking_entities` applied to both local and hosted Supabase. Pricing rules table is empty for now — the engine consuming it lands in Phase 7; Phase 2 only needs the table to exist.
